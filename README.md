# ForgePrompt

**ForgePrompt** is a production-grade AI prompt engineering platform and autonomous agent orchestration engine. It combines a Flask API backend, a parallel DAG workflow execution system, a multi-agent reasoning framework, a community gallery, RAG knowledge bases, and a 3-tier LLM cascade router — all backed by MySQL and optionally a local fine-tuned OMNICORE model running on-device GPU.

Built across 8 development phases, the platform spans everything from basic prompt generation to enterprise-grade saga coordination, event outbox messaging, Kafka integration, and multi-region consensus.

## What It Does

- **Prompt Forging** — Users describe an idea, answer smart MCQ questions generated by the AI, and receive a masterclass-level prompt optimized for their chosen target AI (Midjourney, DALL-E, Cursor, ChatGPT, etc.)
- **Prompt Workbench** — Save, version, fork, rate, and export prompts (TXT / Markdown / JSON)
- **Community Gallery** — Publish prompts publicly, like, fork, follow creators, trending algorithm, Prompt of the Week spotlight
- **Workflow Builder** — Visual DAG workflow editor with 15+ node types (LLM, Search, Condition, Python, SQL, Vision, OCR, KnowledgeBase, Approval, and more)
- **Agent Library** — Define autonomous agents with roles, goals, instructions, preferred models, and tools — then chat with them or embed them in workflows
- **Knowledge Bases** — Upload documents, chunk & embed them, and use them in RAG pipelines inside workflows
- **Organizations** — Team workspaces with RBAC (owner / admin / editor / viewer), secrets manager, and guardrails
- **Marketplace** — Publish and clone workflow templates, rate and review them
- **Admin Panel** — Full platform metrics, user management, moderation, feedback, and analytics
- **Vision-to-Prompt** — Upload an image and reverse-engineer a master prompt from it using Groq vision models
- **OMNICORE** — On-device fine-tuned Qwen2.5 / Phi-3 model via LoRA + Unsloth for fully offline generation

## LLM Cascade System

All LLM calls in `services/llm_service.py` follow a 3-tier cascade with automatic fallback:

```
1. Primary Groq (llama-3.3-70b-versatile)   ← fastest, free tier
         ↓ fails / rate limited
2. OpenRouter (model mapped from config)     ← cloud fallback
         ↓ fails
3. Local OMNICORE (Qwen2.5-3B LoRA, GPU)    ← fully offline
         ↓ generated
4. Refinement pass (OpenRouter Gemini-2.5)  ← polishes local output
```

Both streaming (`call_stream`) and non-streaming (`call`) paths follow this cascade. The active source is reported back in the response metadata.

**Token optimization mode** — set `DEC_TOKEN_USAGE=true` in `.env` to inject a compression directive into every system prompt, reducing token usage at the cost of slightly terse output.

## Multi-Agent System

The agent framework in `agents/` implements the **ReAct** (Reason + Act) pattern:

### Core Components

| Class | File | Responsibility |
|-------|------|---------------|
| `AgentCoordinator` | `coordinator.py` | Delegates tasks to sub-agents via saga transactions |
| `AgentExecutor` | `executor.py` | Manages the run loop, streaming checkpoints, max iteration guard |
| `AgentMemory` | `memory.py` | 4-layer memory: working (token-bounded), episodic, semantic, scratchpad |
| `AgentPlanner` | `planner.py` | Creates goals, decomposes into subgoals, generates ReAct step plans |
| `ReasoningEngine` | `reasoning.py` | Full thought → action → observation loop wired to real tools |
| `ToolRegistryAdapter` | `tools.py` | Plugin-style tool registry with 8 production-ready built-in tools |

### Built-in Agent Tools

| Tool | What it does |
|------|-------------|
| `system_ping` | Returns system health, UTC timestamp, PID |
| `memory_search` | Full-text search across working + semantic memory layers |
| `web_search` | DuckDuckGo scrape — returns up to 5 result snippets |
| `http_fetch` | GET any public URL, returns 2000-char content preview |
| `read_file` | Reads a file from the sandboxed uploads directory |
| `write_file` | Writes text to a file in the sandboxed uploads directory |
| `run_python_sandbox` | Executes Python code via SandboxService (with fallback restricted exec) |
| `llm_generate` | One-shot LLM call via container LLMService or direct LLMService.call() |

### Specialist Agent Roles

- **CodingAgent** — writes and refactors code files via LLM, persists results
- **SecurityAgent** — OWASP vulnerability scanner + STRIDE threat modeller
- **PlannerAgent** — decomposes high-level goals into ordered action lists
- **TestingAgent** — generates unittest files and records execution results

## Workflow Engine

The `WorkflowRunner` in `app.py` executes visual DAG workflows using a `ThreadPoolExecutor` with up to 5 parallel workers.

### Node Types

| Node | Behaviour |
|------|-----------|
| `PromptNode` | Renders a prompt template with variable substitution |
| `PromptRegistryNode` | Pulls a versioned prompt from the Prompt Registry |
| `LLMNode` | Calls the LLM cascade with dynamic routing |
| `AgentNode` | Runs a defined agent with its role/goals/instructions |
| `SearchNode` | DuckDuckGo web search |
| `ConditionNode` | Boolean branch (equals / contains / gt / lt) |
| `DelayNode` | Waits N seconds |
| `ApprovalNode` | Pauses the run for human approval |
| `KnowledgeNode` | RAG similarity search against a knowledge base |
| `OCRNode` | Extracts text from an image/document |
| `VisionNode` | Analyzes an image URL via GPT-4o |
| `SpeechNode` | Transcribes audio |
| `BrowserNode` | Fetches and returns HTTP page content |
| `SQLNode` | Safe read-only SQL queries (SELECT/SHOW/DESC only) |
| `PythonNode` | Executes sandboxed Python code |
| Custom Plugins | Via `PluginSDK.get_plugin(node_type)` |

### Key Features

- **State machine** — valid transitions enforced: `queued → running → completed/failed/paused/cancelled`
- **Retry logic** — per-node configurable retry count + delay
- **Conditional branching** — inactive branches are auto-skipped recursively
- **Streaming checkpoints** — run state persisted to DB so interrupted runs can resume
- **Guardrails** — per-workflow max token and max cost limits
- **Cron scheduler** — lightweight 5-field cron expression evaluator, polls every 30 seconds
- **Webhooks** — any workflow can be triggered via `POST /api/workflows/webhooks/<id>`
- **Import/Export** — full workflow JSON serialization with node ID remapping on import
- **Version snapshots** — every save creates a version; rollback available to any snapshot

## Security

| Layer | Implementation |
|-------|---------------|
| Passwords | bcrypt with per-user salt |
| Sessions | JWT (HS256, 7-day expiry), verified on every protected route |
| Rate limiting | In-memory sliding window: login 5/min, register 3/min, forge 30/min |
| Ownership checks | Every prompt/collection/workflow mutation verifies `user_id` ownership |
| Admin role | Live DB role verification on every admin endpoint (not just JWT claim) |
| Secret encryption | Fernet (AES-128-CBC + HMAC-SHA256) via `services/secret_service.py` |
| SQL injection | 100% parameterized queries, no string interpolation in SQL |
| SQLNode guard | Only SELECT / SHOW / DESC permitted — any write statement is rejected |
| Sandbox | Python code execution in restricted namespace, forbidden token list |
| File I/O sandbox | All agent file tools are path-traversal-protected, scoped to `static/uploads/` |
| Content moderation | Toxic keyword blocklist on community prompt publishing |
| IP anonymization | Analytics use SHA-256 hashed visitor IDs — no raw IPs stored |
| Security log | Dedicated `logs/security.log` for auth failures, ban attempts, bypass tries |
| Request IDs | Every request gets a UUID injected into all log records via `g.request_id` |

## Installation

### Prerequisites

- Python 3.10 or higher
- MySQL 8.0+  (create a database named `forge_db`)
- Git
- *(Optional)* NVIDIA GPU with CUDA for local OMNICORE inference

### 1. Clone and set up the virtual environment

```powershell
git clone <repo-url>
cd prompt

python -m venv env
.\env\Scripts\Activate.ps1

pip install -r requirements.txt
```

### 2. Generate a secret encryption key

This key encrypts organization secrets stored in the database. Generate it once and keep it safe — losing it means stored secrets cannot be decrypted.

```powershell
python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
```

Copy the output into your `.env` file as `SECRET_ENCRYPTION_KEY`.

### 3. Configure the database

Start MySQL and run:

```sql
CREATE DATABASE forge_db CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

The application auto-creates all tables on first boot via `models.init_db()` and `models_phase7.init_phase7_db()`.

### 4. Configure `.env`

See the [Environment Variables](#environment-variables) section below.

### 5. Run the server

```powershell
python app.py
```

The server starts on `http://localhost:5001`.

## Environment Variables

Create a `.env` file in the project root (`d:\Nexabuild\prompt\.env`):

```env
# ── Database ──────────────────────────────────────────────────────────────
MYSQL_HOST=localhost
MYSQL_USER=root
MYSQL_PASSWORD=your_mysql_password
MYSQL_DATABASE=forge_db

# ── Flask ─────────────────────────────────────────────────────────────────
# Generate with: python -c "import secrets; print(secrets.token_hex(32))"
SECRET_KEY=your-secure-random-flask-secret

# ── Secret Encryption (Fernet) ────────────────────────────────────────────
# Generate with: python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
# Required for encrypting organization secrets stored in the database.
SECRET_ENCRYPTION_KEY=your-fernet-key-here

# ── LLM Providers ─────────────────────────────────────────────────────────
GROQ_API_KEY=gsk_your_groq_key_here
GROQ_ACTIVE=true

OPEN_ROUTER_API_KEY=sk-or-v1-your_openrouter_key_here
OPEN_ROUTER_ACTIVE=true

# ── Feature Flags ─────────────────────────────────────────────────────────
# Set to true to compress LLM outputs and reduce token usage.
DEC_TOKEN_USAGE=false

# ── Kafka (optional) ──────────────────────────────────────────────────────
# Only required if using the Kafka event infrastructure.
KAFKA_BOOTSTRAP_SERVERS=localhost:9092
```

### Variable Reference

| Variable | Required | Description |
|----------|----------|-------------|
| `MYSQL_HOST` | Yes | MySQL server hostname |
| `MYSQL_USER` | Yes | MySQL username |
| `MYSQL_PASSWORD` | Yes | MySQL password |
| `MYSQL_DATABASE` | Yes | Database name (default: `forge_db`) |
| `SECRET_KEY` | Yes | Flask session signing key — must be set or app refuses to start |
| `SECRET_ENCRYPTION_KEY` | Recommended | Fernet key for encrypting org secrets at rest. If unset, falls back to base64 (not secure) |
| `GROQ_API_KEY` | Recommended | Primary LLM provider. Free tier available at console.groq.com |
| `GROQ_ACTIVE` | No | Set `true` to enable Groq as Tier 1 LLM provider |
| `OPEN_ROUTER_API_KEY` | Optional | Tier 2 fallback LLM provider |
| `OPEN_ROUTER_ACTIVE` | No | Set `true` to enable OpenRouter fallback |
| `DEC_TOKEN_USAGE` | No | `true` compresses LLM outputs to save tokens |
| `KAFKA_BOOTSTRAP_SERVERS` | Optional | Only needed for Kafka-backed event infrastructure |

## Known Limitations

- **Single-process rate limiter** — `InMemoryRateLimiter` works per-process only. For multi-process deployments behind a load balancer, replace with Redis-backed rate limiting.
- **Thread-pool workflow execution** — `WorkflowRunner` uses Python threads. For CPU-bound node types at scale, migrate to a proper task queue (Celery + Redis or the built-in Kafka execution queue).
- **No test suite** — the codebase does not yet have automated tests. Adding pytest coverage is the next priority.
- **`SECRET_ENCRYPTION_KEY` rotation** — rotating the Fernet key requires re-encrypting all existing `organization_secrets` rows. A migration utility is not yet included.
